# Roadmap Review: Pre-Implementation Brief

**Feature**: `specs/071-nano-service-behaviour/`  
**Roadmap Entry**: `006 - Server: Nano Core Service Behaviour CUs`  
**Created**: 2026-07-14  
**Verdict**: PROCEED WITH UPDATES

## Surfaced Context

Roadmap item 006 targets the service-behaviour CUs left after the type-system and
Base Info slice in roadmap item 005. Its stated outcome is that the server passes
the remaining Core 2022 Server Facet service behaviour requirements, covering
Session change-user, Write with StatusCode/Timestamp and IndexRange, and extended
Discovery/View semantics.

The entry lists these CUs in scope:

- `opc_cu_2317` View TranslateBrowsePath
- `opc_cu_2328` Discovery Get Endpoints
- `opc_cu_2352` Discovery Find Servers Self
- `opc_cu_2389` Attribute Write Values
- `opc_cu_2400` Session Change User
- `opc_cu_2936` Attribute Write StatusCode & Timestamp
- `opc_cu_3147` Attribute Write Index
- `opc_cu_3530` View Basic 2
- `opc_cu_3192` Base Info Diagnostics
- `opc_cu_3983` Base Services Diagnostics

The entry excludes security administration, subscription services, NodeManagement,
Query, and History. It depends on roadmap item 005, now implemented by PR #305 in
`specs/069-nano-type-system/`.

Governance constraints are Constitution Principle I (spec fidelity), Principle IV
(protocol correctness gates), Principle V (security and conformance honesty), and
ADR-0003 as referenced by the roadmap.

## Findings

| Severity | Finding | Evidence | Recommended Action |
|----------|---------|----------|--------------------|
| Must-Address | The roadmap title and outcome overstate nano-default scope. Several in-scope CUs are not `profile_defaults.nano: true` in the manifest. | `opc_cu_2389`, `opc_cu_2400`, `opc_cu_2936`, `opc_cu_3147`, `opc_cu_3192`, and `opc_cu_3983` default off for nano in `profiles/opcua-profile-manifest.yaml`. | Keep the spec scoped to all ten CUs, but require nano claims only where the manifest marks nano default true. Optional and micro+ CUs must stay gated. |
| Must-Address | Existing code appears to satisfy part of the service surface, so implementation tasks must begin with an audit, not blind new behavior. | Existing handlers/tests were found for discovery, view translate, write, session, and diagnostics. | Plan an explicit CU classification step: already satisfied, needs symbol, needs tests, or needs missing behavior. |
| Recommendation | Diagnostics claims need two surfaces: address-space exposure and service-level returned diagnostics. | `tests/unit/test_diagnostics.c` covers live ServerDiagnostics nodes, while `opc_cu_3983` requires available diagnostics through `returnDiagnostics`. | Separate Base Info Diagnostics from Base Services Diagnostics in planning and tasks. |

## Findings Summary

- Must-Address: 2
- Recommendation: 1
- Question: 0

## Recommended Actions

- Proceed to planning with the spec's manifest-first scope boundary.
- Treat roadmap item 006 as a claim-and-gap-closing feature.
- Do not mark optional/micro+ CUs as nano-default claims.
- Apply any roadmap status transition to `in-progress` through the roadmap writer
  workflow if the project wants roadmap status to track this branch.
